# Tier 5 — Config Screen (Demo Script)

**Target length:** 4:30. **Epic:** GRO-gzhu. **UAT:** `.vault/knowledge/uat/uat-GRO-7uiw-tier5-config-screen.md`

## Goal

Show that the operator can configure Groovelab from the KOTS Admin Console
without editing YAML. Demo the embedded-vs-external Postgres toggle, validated
text fields, a generated default that survives upgrade, and a license-gated
config item.

## Setup (do before hitting record)

```bash
# Embedded Cluster VM already provisioned with a Tier 4 install running.
export EC_VM_HOST=user@host
export REPLICATED_API_TOKEN=...

# Pre-warm: confirm the Admin Console is reachable
ssh -L 30880:localhost:30880 $EC_VM_HOST sleep 600 &
open http://localhost:30880

# Browser tabs:
#   1: http://localhost:30880   (Admin Console, logged in)
#   2: Vendor Portal → release/config.yaml view (read-only reference)
#   3: Vendor Portal → customer with track_export_enabled = true
# License file at /tmp/license.yaml carries the track_export entitlement.
```

**If a fresh install is too long for 5 min:** pre-install the VM and walk
the post-install Config Screen. Reserve the install for a longer Loom.

## Script

### 0:00 – 0:20 — Intro

> "Tier 5 — Config Screen. Tier 4 installed Groovelab on a bare VM behind
> KOTS. Tier 5 makes the install configurable: pick embedded or external
> Postgres, set session length, gate guest access, all from the Admin
> Console. No yaml editing, no helm flags."

### 0:20 – 0:50 — release/config.yaml is the contract

Terminal:
```bash
yq '.spec.groups[].name' release/config.yaml
yq '[.spec.groups[].items[].name] | length' release/config.yaml
```

> "Two groups — database and app — and ten items total. The Admin Console
> renders a form from this single file."

Optional one-line callout: `release/helmchart.yaml` maps each config item to
a chart value via `optionalValues` blocks.

### 0:50 – 2:00 — Walk the Config Screen

Browser → Admin Console → **Config**.

Database group:
1. **Database Type** — radio with two options. Default Embedded.
2. Select **External**. Five new fields appear: Host, Port, Username,
   Password, SSL Mode. Switch back to Embedded — the fields hide.
3. Select **External** again. Point at the **Password** field — masked,
   pre-populated with a 24-character generated string. Note: this value
   was generated by the `repl{{ RandomString 24 }}` template at first
   render and KOTS will persist it across upgrades.

Application Settings group:
4. **Session Duration** — type `24` (no unit). Inline validation rejects
   it. Type `24h`. Validation clears.
5. **Max Cards Per Session** — type `abc`. Inline validation rejects it.
   Type `15`. Validation clears.
6. **Enable Guest Access** — toggle, defaults ON.
7. **Enable Track Export** — visible because the test license carries
   `track_export_enabled`. On a license without that field, the item hides.

> "Each item ships with help text. Validation runs in the browser before
> Save. License-gated items only appear when the license entitles them."

### 2:00 – 2:50 — Save and verify the running pod

Click **Save Config**, then **Deploy**. Wait for the new version to roll out.

Terminal:
```bash
ssh $EC_VM_HOST 'kubectl --kubeconfig /var/lib/embedded-cluster/k0s/pki/admin.conf -n groovelab \
  get deploy/groovelab-backend -o yaml' \
  | grep -A1 -E "SESSION_DURATION|MAX_CARDS_PER_SESSION|GUEST_ACCESS_ENABLED"
```

> "The three env vars on the running backend match what we typed in the
> Config Screen. KOTS rendered the helmchart, helm rolled the deployment,
> the values landed on the pod."

### 2:50 – 3:30 — External-DB toggle disables the embedded cluster

Browser → Config → Database Type → **External** (with the fields filled
already), Save, Deploy.

Terminal:
```bash
ssh $EC_VM_HOST 'kubectl --kubeconfig /var/lib/embedded-cluster/k0s/pki/admin.conf -n groovelab \
  get cluster.postgresql.cnpg.io -o name'
# (no output)
```

> "No CNPG Cluster resource. The `db_type=external` choice applied an
> `optionalValues` block in `release/helmchart.yaml` that sets
> `cnpg.createCluster=false`. Same chart, different operator decision."

### 3:30 – 4:10 — Generated password survives upgrade

Browser → Config → External DB Password — show the masked value, click
**Show** briefly, note the first few characters.

Terminal:
```bash
# Cut a no-op release: bump chart version, repackage, promote.
yq -i '.version="0.1.2"' chart/Chart.yaml
helm dependency update chart/
helm package chart/ -d release/
replicated release create --yaml-dir release/ --promote Unstable --app groovelab
rm release/groovelab-*.tgz
```

Admin Console → **Version History** → **Check for updates** → **Deploy**
the new version. After Ready, browser → Config → External DB Password.

> "Same characters. KOTS persisted the saved value across the upgrade
> rather than re-evaluating `RandomString 24` and writing a new password
> into our database connection string."

### 4:10 – 4:30 — Close

> "Two groups, ten items, validated fields, license-gated items, an
> embedded-vs-external toggle that hides irrelevant inputs, and a generated
> default that persists across upgrade. Tier 6 takes this delivery and
> wraps it in a self-serve Enterprise Portal."

## Beats to cut if you run long

- Drop the upgrade-survives-password beat (3:30–4:10) — narrate the
  guarantee instead.
- Drop the env-var verification on the running pod (2:00–2:50) — trust
  the Admin Console.
- Skip the External-DB CNPG-disabled check; voiceover only.
